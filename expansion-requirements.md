# 塾スケジュール管理システム 拡張要件定義書
# 〜AIエージェント・分析・スケジューリング高度化〜

対象リポジトリ: `kankitu-river/juku-system`
技術スタック: Next.js / TypeScript / Supabase (PostgreSQL) / Vercel
本書はClaude Codeへの実装指示のマスタードキュメントである。**フェーズ単位で順に実装**し、一度に複数フェーズを実装しない。

---

## 全体アーキテクチャ方針

### LLM統合
- Anthropic API（Claude）をServer Actions / Route Handlers経由で呼び出す
- APIキーは環境変数 `ANTHROPIC_API_KEY`（Vercelの環境変数に登録、クライアントに絶対露出させない）
- LLM呼び出しは `lib/ai/client.ts` に一元化し、モデル名・共通エラー処理をここで管理
- **原則: AIは提案・下書きまで。DBへの確定書き込みは必ず人間の承認ボタンを経由する**

### データベース
- 新機能はすべて追加テーブル/カラムで実現し、既存テーブルの破壊的変更はしない
- マイグレーションは `supabase/migrations/011_` から連番

### UI
- 既存デザイントークンを踏襲（ネイビー #1E3A5F / アンバー / テール=個別 / パープル=集団 / 赤=不可逆 / 緑=完了 / グレー=非優先）
- AI生成コンテンツには必ず「AI生成」バッジを表示し、人間の確認を促す

# Phase 1: 基盤・整合性（LLM不要／最優先）

**このフェーズはすべてアルゴリズムで実装可能。AI機能の土台となるデータ整合性を固める。**

## 1-1. ダブルブッキング・物理制約の常時検証【33】
- コマの新規作成・更新時（`schedule/actions.ts` の create/update）に以下を検証:
  - 同一生徒が同日同スロットに複数コマ → ブロック（エラー表示）
  - 同一講師が同日同スロットに複数コマ → ブロック
  - 同一ブースが同日同スロットに重複 → 警告（あえて許す運用があり得るため確認ダイアログ）
- 実装: `lib/utils/scheduleValidation.ts` を新設し、Server Action内で保存前に呼ぶ
- 既存データの一括検査ページ `/settings/validation` も追加（違反一覧を表示）

## 1-2. 変更履歴の監査ログ【30】
- 新テーブル `lesson_audit_logs`（id, lesson_id, action['create'|'update'|'delete'], changed_fields jsonb, snapshot jsonb, changed_by, created_at）
- lessons の create/update/delete 時にServer Action内でログをinsert
- コマ詳細画面（`schedule/[id]`）に「変更履歴」タブを追加して時系列表示

## 1-3. 振替候補の自動提案【1】
- 欠席登録時に、条件マッチングで振替候補（講師×日時）をリストアップ
- ロジック（`lib/utils/makeupSuggestion.ts`）:
  1. NG講師を除外（生徒テーブルにng_teacher_ids追加: migration）
  2. 科目対応可能な講師に絞る
  3. 対象期間（2週間先まで）のシフト空きコマに絞る
  4. 相性優先講師（preferred_teacher_ids）を上位表示
  5. 直近の振替履歴で同一講師への偏りがあれば分散
- UI: 振替管理画面に「候補を表示」ボタン → 候補カードから1クリックで仮登録 → 確認 → 確定

## 1-4. 講師の負荷バランス監視【9】
- 週次でコマ数を講師別集計し、平均の1.5倍超の講師をダッシュボードに警告表示
- 実装: ダッシュボードのServer Componentで集計（新テーブル不要）

## 1-5. 休校日・イベントとの衝突検知【10】
- コマ登録時、closures / events テーブルと照合し「この日は◯◯です」と警告表示（登録自体は可能）

## 1-6. 連続コマ・歯抜けシフト検出【28】
- 講師ごとに「出勤日のうち、間に空きコマを挟むパターン」を検出
- シフト管理画面に「歯抜けあり」バッジ、詳細で「第2コマが空いています。詰められる授業: …」を提案

## 1-7. 未確定コマの滞留アラート【36】
- 振替クレジット残があり14日以上未消化の生徒をダッシュボードに警告（既存の振替警告と統合）

## 1-8. 締め切りタスクの自動生成【35】
- 新テーブル `recurring_tasks`（id, title, rule['monthly_day_25'等], lead_days, enabled）
- 設定画面で定型タスク登録（例: 毎月25日「来月シフト確定」）→ ダッシュボードに期日が近いタスクを表示
- タスク完了チェックは `task_completions`（task_id, target_month, completed_at）

## 1-9. キャンセル待ち管理【32】
- 新テーブル `waitlist`（id, student_id, day_of_week, slot_index, subject, created_at, notified_at）
- コマの削除・移動で枠が空いたとき、該当するwaitlistがあればダッシュボードに「空き待ち該当あり」を通知表示

# Phase 2: 分析・可視化（LLM不要）

## 2-1. 座席（ブース）稼働率の分析【26】
- 新ページ `/analytics/booths`: 曜日×スロット別の稼働率（使用ブース数/総ブース数）をヒートマップ表示
- 稼働率90%超のセルは「満席間近」として赤系表示

## 2-2. 曜日×時間帯の混雑ヒートマップ【24】
- 新ページ `/analytics/heatmap`: コマ数ベースの混雑度を色の濃淡で表示（2-1と統合ページでも可）

## 2-3. 講師の指導継続率分析【25】
- 講師×生徒ペアの受講継続期間を集計し、`/analytics/teachers` に継続率ランキングと組み合わせ一覧を表示

## 2-4. 月次レポート自動生成【23】
- 新ページ `/analytics/monthly-report`: 総コマ数・出席率・欠席数・振替消化率・講師別担当数を月次集計
- 既存の印刷インフラ（`@media print`）を使ってA4縦1枚に印刷できるレイアウト
- ※このフェーズでは数値集計のみ。AIコメントはPhase 4で追加

## 2-5. 繁忙期のキャパシティプランニング【29】
- 講習期間の needed（生徒の希望コマ総数） vs available（講師シフトの供給可能コマ総数）を試算
- `/schedule/intensive` に「需給サマリー」カードを追加、不足時は赤字警告

## 2-6. 講師1人あたり同時指導の質的制約【27】
- 設定画面に組み合わせルール登録（例: 「学年差5以上の同時指導は警告」）
- 新テーブル `pairing_rules`（id, rule_type, params jsonb, severity['warn'|'block']）
- コマ保存時のvalidation（1-1の仕組み）に組み込む

## 2-7. ドミノ変更の影響シミュレーション【31】
- 講師の休み登録・コマ削除の前に「影響を確認」ボタン → 影響を受ける生徒一覧・振替候補の有無を表示してから実行
- 1-3の振替候補ロジックを再利用

## 2-8. 学校行事カレンダー連携【34】
- 新テーブル `school_events`（id, school_name, event_type['定期テスト'|'行事'], start_date, end_date）
- 生徒テーブルに school_name カラム追加
- 週間カレンダーに「テスト期間の学校あり」バッジ、該当生徒のコマに小アイコン表示

## 2-9. 来月のシフト予測【11】
- 過去3ヶ月の曜日×スロット別出勤実績から、講師ごとの「出勤しそうな枠」を予測（頻度ベースの単純推定でよい）
- シフトアンケート画面で「前月ベースの予測を初期値としてセット」ボタンを提供

## 2-10. ブース割り当ての自動最適化【12】
- 既存ブース割当に「自動割当」ボタン追加。ルール: 同学年を近接ブースに、集団はまとめる、講師の移動距離最小化
- 貪欲法で十分（厳密最適化は不要）

# Phase 3: LLM導入・チャット/テキスト処理

**このフェーズから `ANTHROPIC_API_KEY` が必要。`lib/ai/client.ts` を最初に作る。**

## 3-0. AI基盤
```
lib/ai/client.ts        … API呼び出しの共通ラッパー（モデル・リトライ・エラー処理）
lib/ai/prompts/         … 機能別プロンプトテンプレート
app/api/ai/[...]/route.ts … ストリーミングが必要なチャット用Route Handler
```
- 個人情報の扱い: プロンプトに含める生徒名は姓のみ or イニシャル化するオプションを設定画面に用意

## 3-1. シフトアンケート自由記述の構造化【3】
- アンケート回答テキスト（「火木の夕方なら行けます」）をLLMでJSON化（曜日・時間帯の配列）
- 抽出結果を人間が確認・修正できるプレビュー画面を挟んでからシフト登録

## 3-2. 自然言語DB検索チャット【4】
- 新ページ `/assistant`: チャットUI
- 実装方式: LLMに「利用可能なクエリ関数一覧」（tool定義）を渡し、function callingで安全な定義済みクエリのみ実行させる
  - 例: getStudentAttendance(student_name, month), getAvailableTeachers(date, slot), getLessonCount(teacher_name, week)
  - **LLMに生SQLを書かせない**（インジェクション・破壊操作防止）
- 回答には参照したデータの根拠（件数・期間）を明記

## 3-3. 週次サマリー自動生成【5】
- 毎週月曜、先週の実績（欠席・振替・特記事項）を集計してLLMで文章化
- ダッシュボードに「今週のサマリー」カードとして表示。生成は手動ボタン起動でよい（cron不要で開始）

## 3-4. 議事録機能【議事録+20+21+22】
- 新テーブル `meeting_notes`（id, date, raw_text, summary, created_at）
- 新テーブル `meeting_tasks`（id, meeting_note_id, description, assignee, due_date, status['open'|'done']）
- 新ページ `/meetings`:
  - テキスト貼り付け（音声入力はスマホのキーボード音声入力を想定、アプリ側では実装しない）
  - LLMで「要約」「決定事項」「TODO（担当・期限つき）」を自動抽出 → 確認画面 → 保存
  - TODOはダッシュボードに未完了件数を表示、完了チェック可能【22】
  - 過去議事録の全文検索（PostgreSQLのilike/全文検索でよい）【21】

## 3-5. 保護者向けお知らせ文の下書き生成【17】（オプション）
- ※本人要望リストには含まれないが3-4と実装がほぼ共通のため、工数僅少で追加可能。不要ならスキップ

## 3-6. 講習提案の根拠生成【15】
- 生徒詳細画面に「講習提案文を生成」ボタン: 出席実績・受講科目・振替状況から保護者説明用の提案文を下書き
- 必ず編集可能なテキストエリアに出力（そのまま送信させない）

## 3-7. 同学年比較の匿名統計【16】
- 生徒詳細に「学年平均との比較」カード（出席率・受講コマ数）。集計はSQL、コメント文生成のみLLM
- 他の生徒個人が特定できる情報は出さない（n<3の学年は「データ不足」と表示）

# Phase 4: 分析高度化・予測（データ蓄積後）

**Phase 1-3の運用で3ヶ月以上データが溜まってから着手する。**

## 4-1. 退塾予兆の検知【8】
- 各生徒の週次出席率の移動平均を計算し、直近4週で有意に低下（例: 20pt以上）した生徒を検出
- まずは統計的ルールベースで実装。将来的にPython+機械学習（異常検知）に置き換え可能な設計にする:
  - 検知ロジックを `lib/analytics/churnDetection.ts` に分離し、インターフェースを固定
  - 将来は同インターフェースの外部API（Python/FastAPI on Render等）に差し替え
- ダッシュボードに「様子見リスト」として表示（「退塾予備軍」等の断定的表現はUIで使わない）

## 4-2. 成績・模試結果の記録と推移分析【13】
- 新テーブル `test_results`（id, student_id, test_name, subject, score, max_score, taken_on）
- 生徒詳細に入力フォーム＋推移グラフ（rechartsは使わずCSS/SVGの簡易グラフでよい。ライブラリ追加する場合は要相談）
- LLMによる所見コメント生成（3-6と同じパターン）

## 4-3. 月次レポートへのAIコメント追加【23の完成】
- 2-4の数値レポートに、前月比較の解説文をLLMで生成して付加

## 4-4. 学習カルテ自動生成【14相当・7】
- 授業notes・出欠・成績の蓄積から生徒ごとのカルテ（強み・弱点・推奨事項）をLLMで生成
- 面談前に「カルテ生成」ボタンで作成、面談資料として印刷可能に

## 4-5. キャパシティ予測の高度化【29の発展・2】
- 講習自動割り振り（既存 intensive/auto）に、2-9のシフト予測と1-3の相性ロジックを統合
